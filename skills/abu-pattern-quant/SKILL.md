# abu-pattern-quant

## 概述

使用这个 skill 把单篇阿布价格行为学文档转换成可落地的量化交付物，包括量化定义、交易策略、检测脚本、策略脚本、运行结果和案例图，而不是停留在文字总结。

适用场景：当用户希望从课程文档、价格行为笔记或形态讲义直接得到量化研究方案、`detect_<pattern>.py`、`strategy_<pattern>.py`、案例图、运行结果或完整产出目录时使用。

默认按以下假设继续，除非用户另有说明：

- 输入是单篇文档，不做目录批处理。
- 输出使用中文，新增文件默认 UTF-8；如果需要 git 提交消息，默认使用中文。
- 数据默认是标准 OHLCV K 线；如果用户提供的是 Binance 原始字段，先按参考约定映射。
- 绘图默认使用 Plotly candlestick 主图叠加。
- 默认创建 `产出/<文档名>/` 目录并落盘交付物；默认同时给出量化定义、交易语义、交易策略、检测脚本、策略脚本、案例图和运行结果摘要。
- 默认样本数据使用 `data/binance_um_perp/ETHUSDT/5m/ETHUSDT-5m-history.parquet`，除非用户指定其他数据。
- 默认将检测逻辑、策略逻辑和绘图逻辑分别保存；不要把所有内容塞进一个文件。

可直接参考仓库中的样本数据：

- `data/binance_um_perp/ETHUSDT/5m/ETHUSDT-5m-history.parquet`

## 工作流

### 1. 读取输入并锁定上下文

- 优先接受 `.txt` 路径或直接粘贴的正文。
- 如果输入是 `.docx` 或 `.pdf`，先提取正文，再继续本 skill 的量化流程；不要在这里重复文档解析细节。
- 只有当缺少关键信息且会实质改变量化定义时才追问，而且优先只问一个高价值问题，例如“这是日内还是日线规则？”。
- 如果没有足够上下文，不要停住；按默认 OHLCV + 中性周期假设继续，并在结果里显式列出假设。
- 进入任务后，先确定本次产出目录：`产出/<文档名>/`；图表放在 `产出/<文档名>/plots/`。

### 2. 抽取形态并归一化到模式族

- 只提炼和量化相关的语义，不复述整篇原文。
- 把自然语言归并到标准模式族，例如趋势、通道、交易区间、突破、双顶双底、三角形、楔形、高潮、支撑阻力、趋势反转。
- 优先打开 `references/abu-pattern-taxonomy.md`，用其中的别名、结构信号和量化抓手做归一化。
- 如果一段文本同时包含“形态识别”和“交易语义”，两者都保留，不要只保留一个。

### 3. 把文字描述翻译成可测特征

- 所有“强”“弱”“明显”“测试”“接近”“回撤”“失败”都必须落成可测条件、阈值或待验证假设。
- 量化定义必须显式写出窗口、比较关系、确认条件、失效条件，以及需要的衍生列。
- 优先使用结构性特征：swing 高低点、触碰次数、斜率、突破幅度、回撤深度、连续 K 线、ATR 比值、通道宽度、收盘位置。
- 如果文档没有给出精确阈值，不能伪造数值；要把它写成“待验证假设”，并给出如何用历史样本校准。

### 4. 输出交易语义与交易策略

- 默认输出：入场触发、止损位置、目标位或测量目标、失败条件、适用品种或周期假设、建议统计指标。
- 如果原文更偏结构识别，也保留最基础的交易语义，不要完全省略。
- 不要把交易语义写成交易建议；要写成可检验的策略假设。
- 在交易语义之后，继续补出最小可执行的交易策略，至少包含：
  - 方向与触发条件
  - 入场规则
  - 出场规则
  - 止损规则
  - 止盈或目标位规则
  - 过滤条件
  - 风险管理假设
  - 建议回测指标
- 默认策略输出应是“可检验策略假设”，不是口头建议；如果原文没有明确给出出场或风控，必须写成待验证假设。

### 5. 生成代码产物

- 默认生成 Python + Polars 1.39 风格代码。
- 必须至少生成以下文件并保存到产出目录：
  - `detect_<pattern>.py`
  - `strategy_<pattern>.py`
  - `plot_<pattern>.py`，或一个可复用绘图脚本
  - `README.md`
  - `run-report.md`
- 检测脚本先做字段准备和列映射，再生成衍生特征列、模式判定列和必要的辅助函数。
- 策略脚本负责把检测列转换成入场、出场、止损、止盈、绩效评价字段；不要把策略逻辑硬塞进 `detect_<pattern>.py`。
- 具体列命名、原始字段映射和实现习惯参考 `references/polars-plotly-conventions.md`。
- 检测脚本默认提供以下接口：
  - `prepare_ohlcv(df)`
  - `add_<pattern>_features(df, ...)`
  - `detect_<pattern>(df, ...)`
  - 如需事后标签，再额外提供 `label_<pattern>_outcomes(df, ...)`
- 策略脚本默认提供 `build_<pattern>_strategy(...)` 或等价入口，用于输出信号、风控和回测评价字段。
- 实时检测与事后标签必须分离：凡是使用识别点之后数据的逻辑，不能混入 `detect_<pattern>` 主函数，必须放入 `label_*` 或明确命名为 outcome/validation 列。
- 生成脚本时，必须在文件头部或函数 docstring 中显式说明哪些列是实时可用，哪些列使用了未来 K 线。

### 6. 运行代码并保存交付物

- 默认运行检测脚本、策略脚本和绘图脚本，不能只写代码不执行。
- 默认生成至少 1 张案例图 `png`；如可行，再生成交互版 `html`。
- 图表至少包含模式区间高亮、关键 swing 点、趋势线或边界线、入场止损目标标记。
- 必须把运行结果写入 `run-report.md`，至少包含：使用的数据文件、样本区间、命中的案例数量、生成的图表路径、是否存在运行失败。
- 如果执行失败，仍要保存文档和代码，并在 `run-report.md` 中记录失败原因、失败步骤和下一步建议。
- 最后在 `README.md` 中汇总：量化定义、交易语义、交易策略、文件清单、运行结果、验证与风险。

## 输出格式

默认先落盘，再在对话中简要汇报交付结果。产出目录和 `README.md` 结构参考 `references/output-contract.md`。

## 执行约束

- 默认使用中文。
- 默认新增或修改文件使用 UTF-8。
- 默认 git 提交消息使用中文。
- 不要把长篇原文拷贝进结果；输出应是压缩后的量化规格和代码。
- 不要为了显得完整而捏造精确阈值；缺失阈值时显式标注假设与校准方法。
- 不要只写单次实验代码；优先写成可复用模块，并给出最小 CLI 或示例入口。
- 产出目录中必须能一眼区分：实时检测、事后标签、策略逻辑、案例图和运行报告。
